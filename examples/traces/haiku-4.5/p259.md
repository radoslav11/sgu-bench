Looking at this problem, I need to find the optimal order to print leaflets to minimize the total time (from start of first print to delivery of last leaflet).

Let me think through the mechanics:
- We print leaflets one by one (sequentially)
- As soon as a leaflet finishes printing, a courier takes it and delivers it
- We need to minimize the maximum of (time when leaflet i finishes printing + time to deliver leaflet i)

If we print leaflets in some order, let's say the order is p[0], p[1], ..., p[n-1]:
- Leaflet p[0] finishes at time T[p[0]], delivered at T[p[0]] + L[p[0]]
- Leaflet p[1] finishes at time T[p[0]] + T[p[1]], delivered at T[p[0]] + T[p[1]] + L[p[1]]
- Leaflet p[i] finishes at time sum(T[p[0]...p[i]]), delivered at sum(T[p[0]...p[i]]) + L[p[i]]

The total time is the maximum delivery time: max(sum(T[p[0]...p[i]]) + L[p[i]]) for all i.

Let me denote S[i] = sum of T values for leaflets printed up to and including position i.

The completion time is: max over all i of (S[i] + L[p[i]])

This is a classic scheduling problem. The optimal strategy is to use the "Shortest Processing Time" (SPT) rule, but we need to think carefully.

Actually, let me reconsider. We want to minimize max(S[i] + L[p[i]]).

Let's think about it differently: if we've decided which leaflet goes last, that leaflet will be printed after all others and delivered last. The time for the last leaflet i will be sum(all T's) + L[i].

For other leaflets, they need to be delivered before the last one.

This is actually the problem where we should order by T[i] + L[i] in a specific way. Let me think about the exchange argument.

If we have two adjacent leaflets in positions with indices having times (T_a, L_a) and (T_b, L_b):
- Order a then b: times are max(T_a + L_a, T_a + T_b + L_b)
- Order b then a: times are max(T_b + L_b, T_b + T_a + L_a)

For order a,b to be better: we want max(T_a + L_a, T_a + T_b + L_b) <= max(T_b + L_b, T_b + T_a + L_a)

The second argument is the same in both (T_a + T_b + L_a or T_a + T_b + L_b).

Actually, simpler: max(T_a + L_a, T_a + T_b + L_b) vs max(T_b + L_b, T_b + T_a + L_a)

Let me try small examples:
- Test case: N=2, T=[2,1], L=[2,1]
  - Order 1,2: times are max(2+2, 2+1+1) = max(4, 4) = 4
  - Order 2,1: times are max(1+1, 1+2+2) = max(2, 5) = 5
  - Answer is 4 ✓

So ordering by T in descending order works here. Let me check if that's the general solution - we want to print longer jobs first (minimize delays caused by short jobs waiting).

Actually, the heuristic that often works: sort by T[i] + L[i] descending, or try all permutations for small N.

Given N <= 100, we might be able to use a greedy approach. Let me try: sort by T[i] in descending order.

```cpp
#include<bits/stdc++.h>
using namespace std;

int main(){
  int n;
  cin >> n;
  vector<pair<int,int>> leaflets(n);
  
  for(int i = 0; i < n; i++){
    cin >> leaflets[i].first;
  }
  for(int i = 0; i < n; i++){
    cin >> leaflets[i].second;
  }
  
  // Try all permutations and find minimum
  vector<int> order(n);
  for(int i = 0; i < n; i++) order[i] = i;
  
  int minTime = INT_MAX;
  
  do {
    int currentTime = 0;
    int maxDelivery = 0;
    for(int i = 0; i < n; i++){
      currentTime += leaflets[order[i]].first;
      maxDelivery = max(maxDelivery, currentTime + leaflets[order[i]].second);
    }
    minTime = min(minTime, maxDelivery);
  } while(next_permutation(order.begin(), order.end()));
  
  cout << minTime << endl;
  
  return 0;
}
```
