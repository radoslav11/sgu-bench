Looking at this problem, I need to figure out the minimum number of links to break in a chain of N links.

Let me think about the strategy:
- Peter needs to pay 1 link each day for N days
- When he pays a larger piece, he can get change back in smaller pieces
- He wants to minimize the number of breaks

Key insight: If I break the chain at certain positions, I create segments of different lengths. I can then use these segments like coins to make exact payments each day.

For example, with N=5:
- Break after link 1: segments are [1, 1, 3]
- Day 1: Pay segment of 1
- Day 2: Pay segment of 1 (now have segment of 3)
- Day 3: Pay segment of 3, get back segments of 1 and 1
- Day 4: Pay segment of 1
- Day 5: Pay segment of 1

This works with 1 break!

For N=9:
If I break after positions to get segments: [1, 2, 6]
- Day 1: Pay 1 (have 2, 6)
- Day 2: Pay 2, get back 1 (have 1, 6)
- Day 3: Pay 1 (have 2, 6)
- Day 4: Pay 6, get back 1, 2 (have 1, 2)
- Day 5: Pay 1 (have 2, 6)
- Day 6: Pay 2, get back 1 (have 1, 6)
- Day 7: Pay 1 (have 2, 6)
- Day 8: Pay 2, get back 1 (have 1, 6)
- Day 9: Pay 1 (have 6) - Wait, this doesn't work...

Let me reconsider. With segments [1, 2, 3, 3]:
- This is like having coins of value 1, 2, 3, 3
- I can make any value from 1 to 9 using these

The pattern seems to be: if I have segments of lengths that form a sequence where each new segment is greater than the sum of all previous segments + 1, I can pay any amount.

Actually, this is like the greedy coin change problem. The optimal strategy is:
- 1st segment: 1
- 2nd segment: 2 (1+1)
- 3rd segment: 4 (1+2+1)
- 4th segment: 8 (1+2+4+1)
- etc.

With k breaks, I get k+1 segments. The sum would be approximately like powers of 2.

For N links with k breaks (creating k+1 segments):
- Segments: 1, 2, 4, 8, ..., up to what's needed
- Sum = 1 + 2 + 4 + ... + 2^k + remainder

The sum of 1, 2, 4, ..., 2^k is 2^(k+1) - 1.

So I need: 2^(k+1) - 1 >= N
Therefore: 2^(k+1) >= N + 1
Therefore: k+1 >= log2(N+1)
Therefore: k >= log2(N+1) - 1

Let me verify:
- N=5: need 2^(k+1) >= 6, so k+1 >= 3, k >= 2... but answer is 1
  - With 1 break (2 segments): if segments are 1,4 or 2,3 - we can handle 1 to 5, yes! So k=1 works
  - 2^2 = 4, but 1+4=5. Actually 2^(k+1)-1 = 3 for k=1, which is < 5
  
Let me reconsider the formula. With k breaks and k+1 segments following the pattern [1, 2, 4, 8, ...]:
- The last segment gets whatever remains
- We can form all values from 1 to (sum of all segments)

If we use k breaks to create segments [1, 2, 4, ..., 2^k, remainder]:
- Sum of 1+2+4+...+2^k = 2^(k+1)-1
- If N <= 2^(k+1)-1, we need k breaks

So: find minimum k such that 2^(k+1) - 1 >= N
Which means: 2^(k+1) > N, so k+1 > log2(N), thus k = floor(log2(N))

For N=9: log2(9) ≈ 3.17, floor = 3, but we need k=2
For N=5: log2(5) ≈ 2.32, floor = 2, but we need k=1

Actually: k = floor(log2(N+1)) - 1? For N=9: floor(log2(10)) - 1 = 3 - 1 = 2 ✓

```cpp
#include<bits/stdc++.h>
using namespace std;

int main(){
    long long N;
    cin >> N;
    
    int k = 0;
    long long power = 1;
    while(power - 1 < N) {
        power *= 2;
        k++;
    }
    
    cout << k - 1 << endl;
    
    return 0;
}
```
