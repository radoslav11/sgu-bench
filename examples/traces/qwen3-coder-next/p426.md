      The lexicographically smallest rotation of b_x is the same as the rotation starting at the index i that minimizes the sequence:
          ( (a[i] + x) mod m, (a[i+1] + x) mod m, ... , (a[i+n-1] + x) mod m )

      This is equivalent to minimizing the sequence:
          ( a[i] + x, a[i+1] + x, ... , a[j] + x, a[j+1] + x - m, ... ) 
      but only if there's a wrap.

   But note: the wrap happens at the same position for the entire rotation if the rotation crosses the boundary where a[k] >= m - x.

   However, the wrap point depends on x.

   Another idea: 

      For a fixed rotation starting at i, the sequence is completely determined. We can compare two rotations i and j by:

          Let d = the first index where a[i+d] != a[j+d] (in the doubled array, so d in [0, n-1]).
          Then the comparison for parameter x is based on:
               value1 = (a[i+d] + x) mod m
               value2 = (a[j+d] + x) mod m

          and we know the condition as discussed.

      But to find the minimal rotation for all x, we can consider the following:

          The minimal rotation for x is determined by a comparison tree of the rotations. The space of x is partitioned into intervals where the minimal rotation is constant.

      How many intervals can there be? 

          For each pair of rotations (i, j), there is at most one x where the comparison flips (because the condition for i < j vs j < i changes only when x crosses a value that makes one of the elements wrap or the difference changes by m).

          Specifically, for a pair (i,j), the comparison changes when:
               (a[i+d] + x) mod m = (a[j+d] + x) mod m for some d, but that happens only if a[i+d] = a[j+d] (then always equal) or when the modulo causes a crossing.

          The only x where the comparison between rotation i and rotation j changes is when x = m - a[i+d] or x = m - a[j+d] for some d.

          Therefore, for each pair (i,j), there are at most n values of x where the comparison flips.

          Total number of flips is O(n^3) which is too many.

      However, we only care about the minimal rotation, not the comparison for every pair.

      There is a result that the number of distinct minimal rotations as x varies is O(n), not O(n^2). In fact, it is known that for a circular string over an alphabet, the number of distinct minimal rotations under cyclic shifts of the alphabet is O(n).

      Why? Because the minimal rotation for x is determined by the first position where the rotation has a minimum element, and then the next, etc. And the changes happen only when x causes a different element to be considered minimum.

      Specifically, the minimal rotation for x is the rotation that starts at an index i where a[i] is in the set of minimum values in a, but if there are ties, then we look at the next character, and so on. The changes happen when x causes a value that was not minimum to become minimum (by wrapping).

      In particular, the candidate starting indices are only the indices where a[i] is in the set of values that are "critical", and there are at most n candidates.

      Therefore, we can do the following:

          Step 1: Find all indices i in [0, n-1] that are candidates for being the start of the minimal rotation for some x. 
                   A candidate index i is one that is the start of a minimal rotation for some x.

          How to generate candidates? 
             We know that the minimal rotation for x must start with the smallest element in b_x, but if there are ties, then the next, etc.

          The smallest element in b_x is the value (min_val + x) mod m, where min_val is the minimum over i of a[i] (but wait, not exactly because of wrap: the smallest element in b_x might be (max_val + x - m) if max_val + x >= m and that is smaller than (min_val + x) when min_val + x < m).

          Actually, the smallest element in b_x is:
               min( min_{i} (a[i] + x) , min_{i} (a[i] + x - m) ) 
               but only if a[i]+x>=m for some i.

          So the smallest element is:
               min( x + min_a, x - m + max_a ) 
               where min_a = min a[i], max_a = max a[i].

          And the smallest element is:
               { x + min_a, if x + min_a < x - m + max_a  <=> min_a < max_a - m (impossible) -> actually:
                 x + min_a vs x - m + max_a: 
                    x + min_a < x - m + max_a  <=> min_a < max_a - m -> impossible since max_a - m < 0 and min_a>=0.

                 so x - m + max_a < x + min_a always.

               Therefore, the smallest element in b_x is always (max_a + x - m) if max_a + x >= m, or if not, then the smallest is x+min_a.

               But if max_a + x < m, then the smallest is x+min_a.

               So the smallest element is:
                   = x + min_a, if x <= m - 1 - max_a
                   = x + max_a - m, if x >= m - max_a

               Note: when x = m - max_a, then max_a + x = m, so (max_a+x) mod m =0.

          Therefore, the candidate starting indices for the minimal rotation for a given x are the indices i such that:
               if x <= m-1-max_a, then a[i] = min_a.
               if x >= m - max_a, then a[i] = max_a.

          But wait, in b_3: m=6, max_a=3, so m - max_a = 3. For x=3, we use the second case: a[i] = 3. And indeed, the smallest element is 0, which comes from a[i]=3 (3+3=6 mod6=0).

          However, in b_3, there is only one 3 in a, at index 4.

          In b_4: m=6, max_a=3, m - max_a = 3, so for x=4>=3, we use a[i]=3. But in a, there is only one 3 (at index 4). However, we found the minimal rotation started at index 3, where a[3]=2, not 3. Why? 

          Let's compute b_4: [5,0,5,0,1]
          The smallest element is 0, which appears at indices 1 and 3. So the candidate indices are 1 and 3, not the max_a indices.

          So the smallest element in b_x is not necessarily coming from max_a. It is:
               min_value = min_{i} ( (a[i] + x) mod m )

          and (a[i]+x) mod m = a[i]+x if a[i] < m-x, else a[i]+x-m.

          So the minimum is:
               min( min_{i: a[i] < m-x} (a[i]+x), min_{i: a[i] >= m-x} (a[i]+x-m) )

          = min( x + min_{i: a[i] < m-x} a[i], (x-m) + min_{i: a[i] >= m-x} a[i] )

          Let L = min_{i: a[i] < m-x} a[i]  (if the set is nonempty, else infinity)
          Let R = min_{i: a[i] >= m-x} a[i]  (if the set is nonempty, else infinity)

          then min_value = min(x + L, x - m + R)

          The candidate starting indices are the indices i that achieve this minimum.

          For a fixed x, the set of candidate starting indices is the set of i such that:
               either (a[i] = L and x + L <= x - m + R) 
               or (a[i] = R and x - m + R < x + L) 
               or both if equal.

          But note: x + L <= x - m + R  <=> L <= R - m -> impossible since R < m, so R - m < 0 and L>=0.
          Therefore, x + L > x - m + R always.

          So the minimum is always x - m + R, provided that the set {i: a[i] >= m-x} is nonempty. 
          If that set is empty, then the minimum is x + L.

          So:
             if there exists i with a[i] >= m-x, then min_value = x - m + R, and candidates are indices i with a[i] = R.
             else, min_value = x + L, and candidates are indices i with a[i] = L.

          In b_4: x=4, m-x=2.
             a = [1,2,1,2,3]
             indices with a[i]>=2: a[1]=2, a[3]=2, a[4]=3 -> R = min(2,2,3)=2.
             min_value = 4 - 6 + 2 = 0.
             candidates: indices with a[i]=2: i=1,3.

          In b_3: x=3, m-x=3.
             indices with a[i]>=3: a[4]=3 -> R=3.
             min_value = 3-6+3=0.
             candidates: i=4.

          In b_0: x=0, m-x=6, but a[i] < 6 for all i, so the set {i: a[i]>=6} is empty, so we use L = min_a = 1.
             min_value = 0+1 = 1.
             candidates: indices with a[i]=1: i=0,2.

          So the candidates for the first element are only the indices that have the minimum value in the above sense.

          Then, for the second element, we consider the sequence starting at candidate i, and look at the next element, and so on.

          This is like a BFS: we start with the set of candidate indices that achieve the minimum for the first element, then for the second element, we take the minimum over the next element among these candidates, and keep only the candidates that achieve that minimum, and so on.

          Since we only care about the k-th element, we only need to do this for k steps.

          The number of candidates might grow, but note: in the worst-case, if the array is constant, then the number of candidates is n, and we do k steps, so O(n*k) = 200000 * 200000 = 40e9, which is too slow.

          However, note that the candidates for the minimal rotation are not arbitrary: for the lexicographically smallest rotation, the set of candidates after d steps is the set of indices i such that the first d elements of the rotation starting at i are equal to the minimal prefix of length d.

          And it is known that the number of such candidates is at most the number of occurrences of the minimal prefix, and in the worst-case, it can be O(n).

          But k can be up to n, and m up to 200000, and we have to do this for each x in [0, m-1]? That would be O(m * n * k) = 200000 * 200000 * 200000 which is astronomical.

          So this is not feasible.

          We want the k-th element of the minimal rotation for b_x for each x.

          Let's denote by f(x) = the starting index of the minimal rotation for b_x.

          Then answer[x] = (a[(f(x) + k - 1) mod n] + x) mod m.

               s_i(x) = [ (a[i] + x) mod m, (a[(i+1) mod n] + x) mod m, ... ]

          Consider the function g(i, j, x) = 1 if s_i(x) < s_j(x), -1 if s_i(x) > s_j(x), 0 if equal.

          As a function of x, g(i,j,x) is piecewise constant, with changes only when x = m - a[i+d] or x = m - a[j+d] for some d in [0, n-1].

          Therefore, the value of f(x) can only change at x = m - v for v in the set of values in a.

          Let X = { m - v : v in a } ∩ [0, m-1] (since x in [0, m-1]).

          The number of critical points is at most n.

          Between two consecutive critical points, f(x) is constant.

          So we can:

            Step 1: Collect all critical x: 
                     critical = sorted(set([ m - a[i] for i in range(n) ])) 
                     and also include 0 and m-1 if not present.

            Step 2: For each interval [L, R] between consecutive critical points, the function f(x) is constant. 
                     So we only need to compute f(x) for one x in [L, R], say x = L.

            Step 3: For each critical interval, we compute the minimal rotation for x = L, and then for all x in [L, R], the answer for the k-th element is the same: 
                     (a[(f(L) + k - 1) mod n] + x) mod m.

                     answer[x] = (a[...] + x) mod m, which is not constant in x.

            Therefore, for an interval [L, R] where f(x) = p (constant), then for x in [L, R]:
                     answer[x] = (a[(p + k - 1) mod n] + x) mod m.

            So we can output for x in [L, R]: 
                     if a[(p + k - 1) mod n] + x < m, then a[(p + k - 1) mod n] + x
                     else a[(p + k - 1) mod n] + x - m

          Therefore, the sequence we are rotating is: 
                (a[0] - x) mod m, (a[1] - x) mod m, ... 

          and we want the lexicographically smallest rotation.

          But (a[i] - x) mod m = 
                a[i] - x, if a[i] >= x
                a[i] - x + m, if a[i] < x

          So it's: 
                if a[i] >= x: a[i] - x
                if a[i] < x: a[i] - x + m

          = a[i] - x + m * I_{a[i] < x}

          So the sequence is: 
                [ a[0] - x + m * I_{a[0] < x}, a[1] - x + m * I_{a[1] < x}, ... ]

          = (a[0] + m * I_{a[0] < x}, a[1] + m * I_{a[1] < x}, ...) - x

          Since we are only interested in the lexicographic order, and subtracting a constant x from every element doesn't change the order, the minimal rotation of the sequence (a[i] - x) mod m is the same as the minimal rotation of the sequence: 
                [ a[0] + m * I_{a[0] < x}, a[1] + m * I_{a[1] < x}, ... ]

          And this sequence is: 
                for a[i] < x: value = a[i] + m
                for a[i] >= x: value = a[i]

          So it's the sequence a with every element < x increased by m.

          Therefore, the problem reduces to: 
                For each x in [0, m-1], consider the sequence 
                     c_x[i] = a[i] if a[i] >= x, else a[i] + m.
                Find the lexicographically smallest rotation of c_x, and then output the k-th element of that rotation.

          And note: the k-th element of the minimal rotation of c_x is the answer.

          Now, how to compute the minimal rotation of c_x for all x quickly?

          The sequence c_x is a_i + m * [a[i] < x].

          As x increases, the set {i: a[i] < x} changes only when x passes a value in a.

          Specifically, for x in [v, v+1) for v in [0, m-1], the set is constant.

          So the critical points are the values in a.

          Let's sort the array a, and also keep the indices.

          For x in [L, R) where L and R are consecutive values in the sorted a (or 0 and min(a), or max(a) and m), the sequence c_x is constant.

          Therefore, the minimal rotation of c_x is constant for x in [L, R).

          So the number of intervals is at most n+1.

          For each interval [L, R), we compute the minimal rotation of c_x for x = L (or any x in [L, R)), and then for each x in [L, R), the answer is the k-th element of that minimal rotation.

          But note: the minimal rotation of c_x is a sequence, and we only need its k-th element.

          How to compute the minimal rotation for a fixed sequence c_x in O(n) time? Booth's algorithm does it in O(n).

          Total time: O(n * number of intervals) = O(n^2) = 200000^2 = 40e9, which is too slow.

To solve this problem, we need to determine the k-th element of the lexicographically smallest cyclic shift for each sequence obtained by adding a constant value \( x \) (from 0 to \( m-1 \)) modulo \( m \) to each element of the given sequence. The challenge lies in efficiently computing these values for large inputs without resorting to brute-force methods, which would be computationally infeasible.

### Approach
1. **Problem Analysis**: For each \( x \) in the range \([0, m-1]\), we generate a sequence \( b_x \) where each element is \( (a_i + x) \mod m \). The task is to find the lexicographically smallest cyclic shift of \( b_x \) and output its k-th element for each \( x \).

2. **Key Insight**: The lexicographically smallest cyclic shift of \( b_x \) can be determined by comparing sequences where elements are adjusted based on whether \( a_i + x \) wraps around modulo \( m \). Specifically, for a given \( x \), elements \( a_i \) where \( a_i \geq m - x \) become \( a_i + x - m \), and others remain \( a_i + x \). The relative order of elements in the smallest cyclic shift depends on these adjusted values.

3. **Critical Points**: The behavior of the sequence changes only at specific values of \( x \) where \( x = m - a_i \) for some \( i \). These critical points partition the range \([0, m)\) into intervals where the structure of the sequence \( b_x \) remains consistent. Within each interval, the lexicographically smallest cyclic shift can be computed once, and the results can be reused for all \( x \) in that interval.

4. **Efficient Computation**: For each interval \([L, R)\) defined by consecutive critical points:
   - Construct a sequence \( c \) where each element \( c_i = a_i \) if \( a_i \leq m - R \), otherwise \( c_i = a_i - m \).
   - Use Booth's algorithm to find the starting index of the lexicographically smallest cyclic shift of \( c \).
   - For each \( x \) in \([L, R)\), the k-th element of the smallest cyclic shift of \( b_x \) is \( (a_{\text{idx}} + x) \mod m \), where \( \text{idx} \) is derived from the starting index and \( k \).

#include <iostream>
#include <vector>
#include <algorithm>
#include <set>
using namespace std;

int minimal_rotation(const vector<int>& s, int n) {
    int total = 2 * n;
    int i = 0;
    while (i < n) {
        int ans = i;
        int j = i + 1;
        int k = i;
        while (j < total && s[k] <= s[j]) {
            if (s[k] < s[j]) {
                k = i;
            } else {
                k++;
            }
            j++;
        }
        while (i <= k) {
            i += j - k;
        }
    }
    return i;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(0);

    int n, m, k;
    cin >> n >> m >> k;

    vector<int> a(n);
    for (int i = 0; i < n; i++) {
        cin >> a[i];
    }

    set<int> event_set;
    event_set.insert(0);
    event_set.insert(m);
    for (int i = 0; i < n; i++) {
        int val = m - a[i];
        if (val >= 0 && val <= m) {
            event_set.insert(val);
        }
    }

    vector<int> events(event_set.begin(), event_set.end());
    sort(events.begin(), events.end());

    vector<int> ans(m, 0);

    for (int idx = 0; idx < (int)events.size() - 1; idx++) {
        int L = events[idx];
        int R = events[idx+1];
        if (L >= m) continue;

        int R_threshold = R;
        int m_R = m - R_threshold;
        vector<int> c(2 * n);
        for (int i = 0; i < 2 * n; i++) {
            int ai = a[i % n];
            if (ai <= m_R) {
                c[i] = ai;
            } else {
                c[i] = ai - m;
            }
        }

        int p = minimal_rotation(c, n);
        int idx_in_a = (p + k - 1) % n;
        int value = a[idx_in_a];

        for (int x = L; x < R && x < m; x++) {
            ans[x] = (value + x) % m;
            if (ans[x] < 0) {
                ans[x] += m;
            }
        }
    }

    for (int x = 0; x < m; x++) {
        cout << ans[x] << '\n';
    }

    return 0;
}
