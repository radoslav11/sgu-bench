After research, I found that the intended solution uses the following insight:

  The function digit_sum(n) has the property that it is periodic with period 10^m for any m, and the average over a period is known. But more importantly, the consecutive digit sums change in a predictable way.

  Specifically, the digit_sum(n+1) = digit_sum(n) + 1 - 9 * (number of trailing 9's in n).

  So if we are at a number n that does not end with many 9's, the digit sum increases by 1.

  In fact, for most numbers, digit_sum(n+1) = digit_sum(n) + 1.

  Only when there is a carry, it drops.

  So the digit_sum function is mostly increasing by 1.

  Therefore, the cumulative digit sum S(n) = S(n-1) + digit_sum(n) = S(n-1) + (digit_sum(n-1) + 1 - 9 * carry(n-1)).

  This might allow us to jump several numbers at once if there is no carry.

  For example, if we are at a number n that ends with d digits that are not 9, then the next d numbers will have digit sums: s, s+1, s+2, ..., s+d.

  So the sum of d consecutive numbers (starting from digit sum s) is d*s + d*(d-1)/2.

  Then we can check if this sum is >= k. If yes, then the passenger will finish within these d numbers. If not, then we can jump d numbers and update the state.

  This is the key to an efficient solution.

 After more research, I found that the correct approach is to use a while loop that jumps over the tickets in blocks of 10, as described, and because the number of blocks is (r-l+1) / 10, and in the worst-case (r-l+1)=10^18, then 10^17 blocks, which is too many.

 unless the number of blocks is not (r-l+1)/10, because the while loop for one passenger might jump over many blocks at once.

 In the block-based approach, for one passenger, the number of blocks it uses is the number of tickets it uses divided by the average block size. Since the block size is at least 1, and the number of tickets per passenger is at most k, then the number of blocks per passenger is at most k. So total number of blocks = (number of passengers) * k.
 number of passengers = (r-l+1) / (average tickets per passenger) = (r-l+1) / (k / average_digit_sum) = (r-l+1) * average_digit_sum / k.
 Then total blocks = [ (r-l+1) * average_digit_sum / k ] * k = (r-l+1) * average_digit_sum.
 average_digit_sum = 81, so total blocks = 81 * (r-l+1) = 8.1e19 for r-l+1=10^18, which is worse.

long long digit_sum(long long n) {
    long long s = 0;
    while (n) {

    }
