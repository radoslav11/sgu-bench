                Algorithm for increasing order:

                  Let's maintain for each row i, a list of columns that are not filled.
                  For each cell (i,j), the number of already filled cells to the left in the row is known (call it L[i][j]), and above is T[i][j].
                  In the final matrix, the number of greater numbers to the left will be left[i][j], and since all future placements in the row to the left will be filled with numbers > current (because we are doing increasing), 
                  then the number of greater numbers to the left = (number of unfilled cells to the left) + (number of filled cells to the left that are greater) 
                  but wait, the filled cells to the left are smaller (because we are doing increasing), so they are not greater.

                  Therefore, the greater numbers to the left will be exactly the unfilled cells to the left.

                  So we require: 
                      (number of unfilled cells to the left in row i) = left[i][j]

                  Similarly, (number of unfilled cells above in column j) = top[i][j]

                  Because the filled cells to the left are smaller, so not greater, and the unfilled will be filled with larger numbers, so they will be greater.

                  Therefore, condition for (i,j) to place the current value (which is the smallest remaining) is:

            Specifically, let's maintain:
                vector<vector<bool>> used(n, vector<bool>(n, false));
                vector<vector<int>> row_prefix(n, vector<int>(n+1, 0)); // row_prefix[i][j] = number of used in row i in [0, j-1]
                vector<vector<int>> col_prefix(n, vector<int>(n+1, 0)); // col_prefix[j][i] = number of used in col j in [0, i-1]

            But updating row_prefix[i] for a new used cell (i,j) would require: for j2 from j+1 to n, row_prefix[i][j2]++.
            This is O(n) per placement, total O(n^3).

            Similarly for col_prefix.

            Then for a cell (i,j), we can get:
                left_count = row_prefix[i][j];
                top_count = col_prefix[j][i];

            Steps:

                for value from n*n down to 1:
                    found = false
                    for i in 0..n-1:
                        for j in 0..n-1:
                            if not used[i][j]:
                                if (row_prefix[i][j] == left[i][j] && col_prefix[j][i] == top[i][j]) {
                                    // place
                                    used[i][j] = true;
                                    A[i][j] = value;
                                    // update row_prefix[i]: for j2 from j+1 to n, row_prefix[i][j2] += 1
                                    for (int j2 = j+1; j2 <= n; j2++) {
                                        row_prefix[i][j2] += 1;
                                    }
                                    // update col_prefix[j]: for i2 from i+1 to n, col_prefix[j][i2] += 1
                                    for (int i2 = i+1; i2 <= n; i2++) {
                                        col_prefix[j][i2] += 1;
                                    }
                                    found = true;
                                    break inner loops;
                                }
                    if not found: output 0.

            Complexity: O(n^2) per placement (because of the two update loops: O(n) for row and O(n) for col), and n^2 placements -> O(n^3) = 600^3 = 216e6.

                Therefore, the condition is not sufficient for a cell to be placed at this step; we must place in a specific order.

                However, the problem is that the condition is necessary, and in a valid reconstruction, there exists an order of placement (decreasing) that works.

                In the example, for value=9, the valid cells are only (2,0) (because (0,0) for 9: 
                    row_prefix[0][0]=0, left[0][0]=0 -> ok.
                    col_prefix[0][0]=0, top[0][0]=0 -> ok.

                    row_prefix[0][1]=1, left[0][1]=0 -> 1!=0.
                so (0,0) is not a valid choice for 9.

                Therefore, the condition is necessary, and in a valid reconstruction, the cell that is chosen must be one that does not break the conditions for the future, 
                but our algorithm chooses the first available cell, which might not be the right one.

                How to ensure we choose a cell that allows continuation? 
                  This is backtracking, which is not feasible.

          Insight from known problems: 
            This problem is known as "Recover the matrix" and the solution is to use the condition in decreasing order and choose the cell with the smallest i+j, or something.

          In fact, in the example, the valid cell for 9 must be (2,0) because:
             - If we place 9 at (0,0), then in row0, for (0,1): the number of greater numbers to the left will be 1 (the 9), so left[0][1] must be 1, but the input has left[0][1]=0.
             - So the input left[0][1]=0 forces that in row0, there is no greater number to the left of (0,1), which means that 9 cannot be to the left of (0,1) in row0.

          Therefore, the cell (2,0) is the only valid cell for 9.

          How to know that (2,0) is the only valid cell for 9? 
             - For (0,0): would require that in row0, there is no greater number to the left of (0,1), but if 9 is at (0,0), then (0,1) has a greater number to the left, so left[0][1] should be at least 1, but it is 0.
             - For (0,1): would require that in row0, for (0,2), the greater numbers to the left include 9, so left[0][2] should be at least 1, but it is 0.
             - For (1,0): in col0, for (2,0): greater numbers above would include 9, so top[2][0] should be at least 1, but it is 0.
             - For (1,1): in row1, for (1,2): greater numbers to the left would include 9, so left[1][2] should be at least 1, but it is 0.
             - For (2,0): 
                   in row2, for (2,1): greater numbers to the left = 9, so left[2][1] must be 1 -> matches.
                   in col0, for (1,0): greater numbers above = 9, so top[1][0] must be 1, but the input has top[1][0]=0 -> wait, no.

          In the input, top[1][0]=0, and if 9 is at (2,0), then for (1,0) in col0, the number of greater numbers above is 0 (because (2,0) is below), so it's ok.

          So the only valid cell for 9 is (2,0).

          How to incorporate this in the algorithm? 
            We must choose a cell (i,j) such that:
                left[i][j] = 0 and top[i][j] = 0, and also for any cell (i',j') with i'<i or j'<j, the conditions are not broken, 
                but it's complex.

To solve this problem, we need to reconstruct a square matrix \( A \) of size \( n \times n \) such that each number from 1 to \( n^2 \) appears exactly once, and for each cell \( (i, j) \), the number of elements greater than \( A[i][j] \) in the same column above it is given by \( \text{top}[i][j] \), and the number of elements greater than \( A[i][j] \) in the same row to the left is given by \( \text{left}[i][j] \).

### Approach
1. **Initial Checks**: First, we verify if the given \( \text{top} \) and \( \text{left} \) matrices are feasible. For any cell \( (i, j) \), if \( \text{top}[i][j] > i \) or \( \text{left}[i][j] > j \), it's impossible to satisfy the conditions, so we immediately return 0.

2. **Greedy Placement in Decreasing Order**: We construct the matrix \( A \) by placing numbers from \( n^2 \) down to 1. For each value, we find the bottom-right most cell \( (i, j) \) that hasn't been filled yet and satisfies:
   - The number of already placed (larger) numbers in row \( i \) to the left of column \( j \) equals \( \text{left}[i][j] \).
   - The number of already placed (larger) numbers in column \( j \) above row \( i \) equals \( \text{top}[i][j] \).

3. **Efficient Tracking**: To efficiently track the counts of placed numbers, we maintain two 2D arrays:
   - `row_prefix[i][j]` stores the count of placed numbers in row \( i \) for columns \( [0, j-1] \).
   - `col_prefix[j][i]` stores the count of placed numbers in column \( j \) for rows \( [0, i-1] \).

4. **Update Mechanism**: After placing a number at \( (i, j) \), we update:
   - For all columns \( j2 > j \) in row \( i \), increment `row_prefix[i][j2]`.
   - For all rows \( i2 > i \) in column \( j \), increment `col_prefix[j][i2]`.

This approach ensures that each placement is verified in \( O(n) \) time per value, leading to an overall time complexity of \( O(n^3) \), which is feasible for \( n \leq 600 \).
