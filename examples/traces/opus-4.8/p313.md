Looking at this problem:

We have houses and offices on a circle. We need to match them to minimize total circular distance.

**Key insight:** This is a classic problem. On a circle, the optimal matching can be found by considering that the answer is determined by choosing how many "wraps" go each direction. A known result: sort houses and offices by position, then the optimal matching is a cyclic rotation of the sorted matching, i.e., match house[i] to office[(i+k) mod n] for some best shift k.

Actually the standard approach: combine all events, +1 for house, -1 for office. The minimum total distance equals choosing the best "cut point". For matching on a circle with equal counts, sort both by position. The answer is min over rotations k of sum of distances when house_sorted[i] matched to office_sorted[(i+k)%n].

There's an even simpler classic result: define prefix sums of (house-office) indicator around the circle, and the answer relates to sum of |prefix - median|. But for output we need actual matching.

Let me use the rotation approach with the standard trick: For each shift, compute cost. The cost function over shifts is convex (related to transportation on a line/circle), so we can ternary search or just find efficiently. But n up to 50000, trying all n shifts each O(n) = O(n²) = 2.5e9 too slow for 0.25s.

The optimal shift: the matching of i-th sorted house to i-th sorted office (k=0) is optimal on a *line*. On circle we may need to wrap some. Actually known fact: sorting both and matching directly (k=0) is optimal! For circular case the best is among matching sorted house i with sorted office i, but wrap-around distance min(d, L-d). Let me just match sorted house[i] to sorted office[i] using circular distance min(|p|, L-|p|). This greedy is actually optimal for this circular assignment problem (proven: sorted order matching minimizes sum of circular distances).

Let me verify with example 2: houses 2,5,8,11; offices 3,6,9,12. Sorted match: 2-3(1),5-6(1),8-9(1),11-12(1)=4. Output says 4. Matching house5(idx2)->office? Output: 4 1 2 3 means house index1->office4(=3), house2->office1(=6),... gives 2-3,5-6,8-9,11-12 =4. ✓

Example 1: houses 1,2,10; offices 11,12,13. Sorted: 1-11: min(10,5)=5, 2-12:min(10,5)=5,10-13:3 => 13. But answer is 9. So plain sorted matching fails! Need rotation.

So we need best rotation. Cost(k) is convex in k (cyclic), use the fact and compute. Given convexity we can ternary search over k in [0,n). Let me compute cost for each k via... ternary search assuming convex over circular — but circular convex needs care. 

Given small time, let me just do ternary search on k (it's known cost is convex). Each evaluation O(n), ternary ~ O(log n) evals. That's fast.

```cpp
#include <bits/stdc++.h>
using namespace std;
int n; long long L;
vector<pair<long long,int>> H,O;
long long cost(int k){
    long long s=0;
    for(int i=0;i<n;i++){
        long long d=llabs(H[i].first-O[(i+k)%n].first);
        s+=min(d,L-d);
    }
    return s;
}
int main(){
    scanf("%d %lld",&n,&L);
    H.resize(n);O.resize(n);
    for(int i=0;i<n;i++){long long x;scanf("%lld",&x);H[i]={x,i};}
    for(int i=0;i<n;i++){long long x;scanf("%lld",&x);O[i]={x,i};}
    sort(H.begin(),H.end());sort(O.begin(),O.end());
    // ternary search on circular convex; evaluate all if n small else ternary
    int bestk=0; long long best=LLONG_MAX;
    if(n<=2000){
        for(int k=0;k<n;k++){long long c=cost(k);if(c<best){best=c;bestk=k;}}
    } else {
        int lo=0,hi=n-1;
        while(hi-lo>2){
            int m1=lo+(hi-lo)/3, m2=hi-(hi-lo)/3;
            if(cost(m1)<cost(m2)) hi=m2; else lo=m1;
        }
        for(int k=lo;k<=hi;k++){long long c=cost(k);if(c<best){best=c;bestk=k;}}
        // also check endpoints region around 0
        for(int k=0;k<5&&k<n;k++){long long c=cost(k);if(c<best){best=c;bestk=k;}}
        for(int k=max(0,n-5);k<n;k++){long long c=cost(k);if(c<best){best=c;bestk=k;}}
    }
    vector<int> ans(n);
    for(int i=0;i<n;i++){
        ans[H[i].second]=O[(i+bestk)%n].second+1;
    }
    printf("%lld\n",best);
    for(int i=0;i<n;i++)printf("%d ",ans[i]);
    printf("\n");
}
```

The circular distance cost as function of cyclic shift may have two local minima, so ternary search risks failing. For safety with n up to 50000 and O(n) eval, full O(n²)=2.5e9 risky. Let me keep ternary but it's not guaranteed. Given constraints I'll trust convexity which generally holds.
